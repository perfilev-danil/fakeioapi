1. Создаем Express-приложение

const app = express();

* Что это: express() создаёт серверное приложение на Node.js.
* Для чего: это основной объект, через который мы добавляем маршруты (routes), middleware и обрабатываем HTTP-запросы.
* Пример: когда кто-то откроет в браузере [http://localhost:3000/metrics](http://localhost:3000/metrics), Express поймает этот запрос и ответит.


2. Middleware для парсинга данных

app.use(express.json());
app.use(express.urlencoded({ extended: true }));

* express.json() – позволяет серверу читать данные из тела запроса в формате JSON.
* express.urlencoded(...) – позволяет читать данные из форм (например, HTML формы, которые отправляются через POST).
* Зачем: без этого сервер не сможет понять, что вы присылаете, например, { "email": "[a@b.com](mailto:a@b.com)" } в теле POST-запроса.


3. Подключаем REST-маршруты

app.use("/auth", authRoutes);
app.use("/metrics", metricRoutes);

* Что это: мы говорим серверу: “всё, что начинается с /auth, обрабатывает модуль authRoutes, а всё, что начинается с /metrics, — metricRoutes”.
* Зачем: так код становится структурированным. В одном месте auth (регистрация, логин), в другом — метрики.
* Пример:

  * POST /auth/register → регистрация нового пользователя
  * GET /metrics → получить последние 100 метрик


4. Создаем HTTP-сервер для WebSocket

const server = http.createServer(app);

* Что это: Node.js умеет создавать сервер “сам по себе”, Express только добавляет удобную оболочку.
* Зачем: WebSocket (Socket.IO) не работает напрямую на объекте app, ему нужен настоящий HTTP-сервер.
* Пример: сервер одновременно может принимать обычные HTTP-запросы и WebSocket-соединения для live-данных.


5. Инициализация WebSocket (Socket.IO)

const io = initSocket(server);

* Что это: создаём сервер WebSocket через Socket.IO.
* Зачем: WebSocket позволяет серверу отправлять данные клиентам мгновенно, без того чтобы клиент постоянно спрашивал “есть ли новые данные?” (как в обычном HTTP).
* Пример: график на сайте обновляется каждую секунду без перезагрузки страницы.
* Что делает initSocket:

  1. Подключает Socket.IO к HTTP-серверу.
  2. Слушает новые подключения клиентов (connection).
  3. Может отправлять события (emit) всем подключенным клиентам.


6. Поток метрик (startMetricStream)

startMetricStream(io);

* Что это: функция, которая каждые 1 секунду:

  1. Создаёт фейковую метрику (generateFakeMetric)
  2. Сохраняет её в базу данных (Prisma + PostgreSQL)
  3. Отправляет всем подключённым клиентам через Socket.IO (metric:update)
* Зачем: чтобы на сайте или приложении был живой поток данных, имитирующий, например, котировки криптовалют.
* Почему это нужно: без этого пользователь видел бы только старые данные из базы.


7. REST endpoints для метрик

* GET /metrics → возвращает последние метрики из базы, чтобы можно было показать историю.
* GET /metrics/:id → возвращает конкретную метрику по ID.
* Зачем: WebSocket — это поток, но иногда нужно просто “получить старые данные” через обычный HTTP-запрос.
* Это удобно, например, когда клиент только подключился и хочет сразу показать последние значения на графике.


8. Почему мы используем одну функцию для генерации метрик

createAndSendMetric()

* Что делает: создаёт метрику, сохраняет её в базу и рассылает всем клиентам.
* Почему так:

  1. Нет дублирования кода.
  2. Поток (автоматически каждую секунду) и тестовый endpoint используют одну и ту же функцию.
  3. Легко менять генерацию метрик в одном месте, и всё работает одинаково.


9. Запуск сервера

const PORT = process.env.PORT || 3000;
server.listen(PORT, () => console.log(`Server running on port ${PORT}`));

* Что делает: запускает сервер, чтобы он принимал HTTP-запросы и WebSocket-соединения.
* Почему server, а не app.listen: потому что WebSocket подключается к HTTP-серверу, а не к Express напрямую.
* Вывод в консоль: удобно видеть, что сервер стартовал.


10. Как всё работает вместе


11. Сервер стартует → подключается к базе данных → создаёт HTTP + WebSocket сервер.


12. Клиенты открывают сайт → подписываются на события metric:update через WebSocket.


13. startMetricStream каждые 1 секунду:

    * генерирует метрику,
    * сохраняет её в базу,
    * рассылает её клиентам.


14. Пользователь видит живые обновления на графике.


15. Если нужно историю → делает HTTP GET /metrics и получает последние метрики.
